# Scope Management

> "One of the biggest threats to a project is not running out of money—it's slowly agreeing to do more than was originally promised."

---

# Learning Objectives

After completing this lesson you should be able to:

- Define project scope.
- Distinguish product scope from project scope.
- Explain why scope management is essential.
- Understand Scope Creep.
- Explain how a Work Breakdown Structure (WBS) is developed.
- Evaluate requests for scope changes.
- Balance scope with time, cost and quality.

---

# What is Scope?

Project scope defines **exactly what the project will deliver—and equally important—what it will NOT deliver.**

Many failed projects did not fail because the team lacked technical competence.

They failed because nobody clearly defined the project's boundaries.

If scope is unclear:

- stakeholders make assumptions,
- customers request "small" additions,
- priorities change,
- budgets increase,
- schedules slip.

A well-defined scope provides a common understanding between everyone involved.

---

# Product Scope vs Project Scope

Students often confuse these concepts.

## Product Scope

Describes the characteristics and functions of the final deliverable.

Example:

A warehouse management system should:

- track inventory
- support barcode scanners
- generate reports
- integrate with ERP

These features describe the **product**.

---

## Project Scope

Describes all work required to deliver that product.

For example:

- requirements gathering
- software development
- testing
- user training
- deployment
- documentation

This is the **project scope**.

---

# Why Scope Matters

Imagine building a new container terminal.

Initially the project includes:

- one quay
- two cranes
- administrative building

Halfway through construction someone says:

> "Could we also add an automated gate system?"

A month later:

> "Maybe include a rail connection."

Later:

> "Can we redesign the warehouse?"

Each request may seem reasonable individually.

Together they fundamentally change the project.

---

# Scope Creep

This phenomenon is called **Scope Creep**.

Scope Creep is the uncontrolled expansion of project scope without corresponding adjustments to:

- budget
- schedule
- resources

It is one of the leading causes of project overruns.

Scope creep usually happens gradually.

Nobody notices until deadlines begin slipping.

---

# Causes of Scope Creep

Common causes include:

- unclear requirements
- weak project governance
- poor stakeholder management
- optimistic planning
- changing customer expectations
- technological developments
- poor documentation

Notice that technical problems are often **not** the primary cause.

People are.

---

# Work Breakdown Structure (WBS)

The most common tool for defining project scope is the Work Breakdown Structure.

The WBS breaks a project into increasingly smaller deliverables.

Example:

Warehouse Project

1. Planning
    - Requirements
    - Feasibility
2. Construction
    - Foundation
    - Steel Structure
    - Roof
3. Automation
    - Conveyor System
    - Software
4. Testing
5. Commissioning

Notice that the WBS focuses on **deliverables**, not chronological activities.

---

# The 100% Rule

A professional WBS follows the **100% Rule**.

This means:

Every deliverable required to complete the project is included.

Nothing extra is included.

The total scope equals exactly 100%.

No more.

No less.

---

# Scope Baseline

Once approved, project scope becomes the **Scope Baseline**.

The baseline represents the official agreement between stakeholders.

Every future change is measured against this baseline.

Without a baseline, there is no objective way to determine whether scope has changed.

---

# Scope Change Control

Should scope ever change?

Yes.

Projects operate in changing environments.

However, changes should follow a structured process.

Typical questions include:

- Why is the change needed?
- What benefits does it create?
- What risks does it introduce?
- How will it affect cost?
- How will it affect schedule?
- What happens if we reject it?

Professional organizations never approve changes simply because someone requests them.

---

# Case Study

A hospital is constructing a new emergency department.

Two months before completion surgeons request:

- larger operating rooms
- additional MRI facilities
- expanded parking

All requests create value.

However:

Budget is fixed.

Completion date is legally mandated.

The project manager cannot simply approve every request.

Instead, each change must be evaluated through formal change control.

---

# Common Mistakes

❌ Starting work before defining scope.

❌ Assuming stakeholders have the same understanding.

❌ Accepting "small" requests without documentation.

❌ Creating a WBS based on departments instead of deliverables.

❌ Treating scope changes as free.

---

# Key Concepts

| Concept | Meaning |
|---------|---------|
| Scope | The agreed work and deliverables of the project. |
| Product Scope | Features of the final product. |
| Project Scope | Work required to deliver the product. |
| Scope Creep | Uncontrolled growth of project scope. |
| WBS | Hierarchical decomposition of deliverables. |
| Scope Baseline | Approved version of project scope. |

---

# Critical Thinking

Excellent project managers understand that saying **"No"** is sometimes the most valuable decision they make.

Every additional feature has an opportunity cost.

Accepting one request often means sacrificing:

- budget
- schedule
- quality
- or another feature.

Managing scope is therefore largely about making informed trade-offs.

---

# Summary

Successful projects clearly define:

- what will be delivered,
- what will not,
- who approves changes,
- how changes are evaluated.

Remember:

> **The biggest danger is rarely a large scope change.**
>
> **It is hundreds of small ones.**
